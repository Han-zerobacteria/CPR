# CPR 로그인 프론트 페이지 요구사항 정의서

날짜: 2026-05-21

## 1. 목적

CPR 서비스의 사용자가 아이디와 비밀번호를 입력하여 허가된 사용자인지 인증받는 로그인 페이지를 구현한다.

로그인 성공 시 프론트엔드 인증 상태를 구성하고, 서비스 기본 화면으로 이동할 수 있어야 한다.

---

## 2. 참고

프로젝트 구조 및 개발 규칙은 `AGENTS.md`를 따른다.

---

## 3. 페이지 경로

```txt
/login
```

---

## 4. 핵심 UI/UX 방향

- 모바일 우선 반응형 UI로 구현한다.
- 추후 PWA 적용을 고려하여 모바일 환경에서도 자연스럽게 동작해야 한다.
- CPR 브랜드 primary color는 black 계열로 사용한다.
- 화면 전체를 어둡게 만드는 것이 아니라, 주요 CTA, 강조 텍스트, 포커스 상태에 black을 적용한다.
- 입력 오류 및 성공 여부를 명확하게 표시한다.
- 사용자는 로그인 요청 중 상태를 즉시 인지할 수 있어야 한다.
- 작은 화면과 모바일 키보드 노출 상황에서도 입력 및 제출이 자연스럽게 가능해야 한다.

---

## 5. 입력 필드

| 필드     | 설명                   | 필수 여부 |
| -------- | ---------------------- | --------- |
| login_id | 로그인에 사용할 아이디 | 필수      |
| password | 비밀번호               | 필수      |

---

## 6. 유효성 검증 요구사항

## 6-1. login_id

- 필수 입력
- 공백만 입력 불가
- 미입력 시 에러 메시지를 입력창 하단에 표시한다.

예시:

```txt
아이디를 입력해주세요.
```

---

## 6-2. password

- 필수 입력
- 입력값은 마스킹 처리한다.
- 미입력 시 에러 메시지를 입력창 하단에 표시한다.

예시:

```txt
비밀번호를 입력해주세요.
```

---

## 7. UI 요구사항

## 7-1. 반응형 요구사항

- 모바일 우선 반응형 구조
- PWA 확장을 고려한 UI 설계
- 작은 화면에서도 입력이 자연스럽게 가능해야 함
- 모바일 키보드 노출 시 레이아웃 깨짐 최소화
- 버튼 및 입력 영역은 터치 친화적 크기 유지

---

## 7-2. 컬러 정책

### 브랜드 컬러

- CPR 브랜드 primary color는 black 계열이다.
- 브랜드 primary color는 주로 로그인 버튼, 제목, 링크, 입력 포커스 상태에 사용한다.
- 가시성을 위해 폼 surface는 밝은 배경을 유지한다.

### 상태 컬러

| 상태     | 방향                         |
| -------- | ---------------------------- |
| 기본     | 밝은 surface, black CTA      |
| 포커스   | black border 또는 focus ring |
| 실패     | red 계열                     |
| disabled | black CTA 유지, opacity 조정 |

---

## 7-3. 폼 구성

- 로그인 제목 표시
- 입력 폼 중앙 정렬
- 각 필드에 label 제공
- 필수 입력값 표시
- 에러 메시지는 입력창 하단 표시
- 로그인 실패 메시지는 폼 내부에 별도 영역으로 표시

---

## 7-4. 버튼

### 로그인 버튼

- 필수 입력값이 모두 유효할 때 제출 가능
- 요청 중 비활성화하여 반복 요청을 방지한다.
- 요청 중 로딩 텍스트를 표시할 수 있다.
- 버튼은 활성/비활성 상태 모두 화면에서 명확히 보여야 한다.
- 브랜드 primary black을 기본 배경으로 사용한다.

### 회원가입 이동 버튼

- 회원가입 페이지 이동 링크를 제공한다.

```txt
/signup
```

---

## 8. 성공 시 동작

- 로그인 API 요청 성공 후 access token을 프론트엔드 인증 저장소에 저장한다.
- 로그인 응답에는 프로필 정보가 없으므로 `/api/auth/me/`를 추가 호출하여 nickname 및 profile image 정보를 조회한다.
- 프로필 조회가 완료되면 전역 인증 세션을 구성한다.
- 기본값 `/`로 이동한다.

---

## 9. 에러 메시지

인증 실패 시 계정 존재 여부를 노출하지 않기 위해 통합 메시지를 사용한다.

```txt
아이디 및 비밀번호가 잘못 되었습니다.
아이디와 비밀번호를 정확히 입력해주세요
```

프로필 정보 조회 실패 시에는 로그인 정보를 완성할 수 없으므로 별도 오류를 표시한다.

```txt
로그인 정보를 불러오지 못했습니다. 잠시 후 다시 시도해주세요.
```

---

## 10. 보안 및 UX 고려사항

- 로그인 페이지에서는 아이디 중복 검사 또는 계정 존재 여부 확인 API를 호출하지 않는다.
- 로그인 실패 사유를 아이디/비밀번호 단위로 세분화하지 않는다.
- access token은 현재 프로젝트 정책에 따라 메모리 기반 auth store에 저장한다.
- refresh token은 백엔드의 httpOnly cookie 정책을 따른다.
- 비밀번호 입력값은 로그, 문서, 화면에 평문으로 노출하지 않는다.

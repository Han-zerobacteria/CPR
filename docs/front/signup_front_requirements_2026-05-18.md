# CPR 회원가입 프론트 페이지 요구사항 정의서

## 25/05/13

## 1. 목적

CPR 서비스의 신규 사용자가 계정을 생성하고, 기본 프로필 정보를 등록할 수 있는 회원가입 페이지를 구현한다.

회원가입은 단순 계정 생성뿐 아니라 이후 프로필 조회, 코디 요청, 콘텐츠 활동에 필요한 기본 사용자 정보를 함께 수집하는 것을 목표로 한다.

---

# 2. 참고

## 프로젝트 구조 및 개발 규칙은 `AGENT_GUIDE.md`를 따른다.

# 3. 페이지 경로

```txt
/signup
```

---

# 4. 핵심 UI/UX 방향

- 모바일 우선 반응형 UI로 구현한다.
- 추후 PWA 적용을 고려하여 모바일 환경에서도 자연스럽게 동작해야 한다.
- 기본 컬러는 검은색 계열을 사용한다.
- 별도의 중복 확인 버튼 없이 자동 중복 검사를 수행한다.
- 사용자는 입력 결과를 즉시 시각적으로 인지할 수 있어야 한다.
- 입력 오류 및 성공 여부를 명확하게 표시한다.

---

# 5. 입력 필드

| 필드             | 설명                   | 필수 여부 |
| ---------------- | ---------------------- | --------- |
| login_id         | 로그인에 사용할 아이디 | 필수      |
| password         | 비밀번호               | 필수      |
| confirm_password | 비밀번호 확인          | 필수      |
| nickname         | 프로필에 표시될 닉네임 | 필수      |
| profile_image    | 프로필 이미지          | 선택      |
| bio              | 자기소개               | 선택      |

---

# 6. 유효성 검증 요구사항

## 6-1. login_id

- 필수 입력
- 공백 입력 불가
- 최소 4자 이상
- 영문, 숫자 조합 허용
- 입력칸 포커스 해제(onBlur) 시 자동 중복 검사 수행
- 기본 유효성 검증 실패 시 중복 검사 API 호출 금지

---

## 6-2. password

- 필수 입력
- 최소 8자 이상
- 영문 및 숫자 포함 권장
- 입력값은 기본적으로 마스킹 처리
- 공백 포함 여부는 백엔드 정책과 동일하게 처리

---

## 6-3. confirm_password

- 필수 입력
- password와 동일해야 함
- 불일치 시 제출 불가

---

## 6-4. nickname

- 필수 입력
- 최소 2자 이상
- 공백만 입력 불가
- 입력칸 포커스 해제(onBlur) 시 자동 중복 검사 수행
- 기본 유효성 검증 실패 시 중복 검사 API 호출 금지

---

## 6-5. profile_image

- 선택 입력
- 허용 확장자:
  - jpg
  - jpeg
  - png
  - webp
  - avif

- 잘못된 확장자 업로드 시 에러 메시지 표시
- 업로드 전 미리보기 제공
- 이미지 미선택 시 기본 프로필 이미지 사용

---

## 6-6. bio

- 선택 입력
- 최대 30자
- 현재 글자 수 표시 가능
- 최대 길이 초과 시 입력 제한 또는 에러 표시

---

# 7. 중복 검사 요구사항

## 7-1. 공통 요구사항

- login_id 및 nickname 모두 자동 중복 검사 적용
- 별도의 "중복 확인" 버튼 사용 금지
- 입력칸 포커스 해제(onBlur) 시 자동 검사 수행
- API 요청 중 중복 요청 방지
- 검사 완료 후 값을 수정하면 기존 검사 상태 초기화

---

## 7-2. 성공 상태

중복 검사 성공 시:

- 입력 필드 테두리를 초록색으로 변경
- 성공 메시지 표시

예시:

```txt
사용 가능한 아이디입니다.
사용 가능한 닉네임입니다.
```

---

## 7-3. 실패 상태

중복 검사 실패 시:

- 입력 필드 테두리를 빨간색으로 변경
- 실패 사유를 텍스트로 표시
- 회원가입 제출 불가

예시:

```txt
이미 사용 중인 아이디입니다.
이미 사용 중인 닉네임입니다.
아이디는 최소 4자 이상이어야 합니다.
닉네임은 최소 2자 이상이어야 합니다.
영문과 숫자만 사용할 수 있습니다.
```

---

## 7-4. 기본 상태

- 아직 검사를 수행하지 않았을 경우 기본 검은색 테두리 유지
- 입력값 수정 시 success/error 상태 즉시 초기화

---

## 7-5. 로딩 상태

- 중복 검사 API 요청 중 로딩 상태 표시
- 로딩 중에는 중립 테두리 유지
- 로딩 중 추가 요청 방지

---

## 7-6. 입력 필드 상태 우선순위

```txt
로딩 중 > 에러 상태 > 성공 상태 > 기본 상태
```

# 9. 회원가입 제출 가능 조건

회원가입 버튼은 아래 조건을 모두 만족할 때만 활성화한다.

- login_id 유효성 검증 통과
- login_id 중복 검사 통과
- password 유효성 검증 통과
- confirm_password 일치
- nickname 유효성 검증 통과
- nickname 중복 검사 통과
- profile_image 형식 검증 통과
- bio 최대 길이 검증 통과
- API 요청 중이 아님

---

# 10. 회원가입 요청 흐름

1. 사용자가 회원가입 정보를 입력한다.
2. 프론트에서 기본 유효성 검증 수행
3. 실패 시 API 요청 금지
4. 검증 통과 시 회원가입 API 호출
5. profile_image 존재 시 multipart/form-data로 함께 전송
6. 회원가입 성공 시 로그인 처리
7. 실패 시 서버 에러 메시지 표시

---

# 11. API 연동 요구사항

## 11-1. 회원가입 API

### 요청

```txt
POST /api/accounts/signup/
```

### Content-Type

```txt
multipart/form-data
```

### Request Body

```txt
login_id
password
confirm_password
nickname
profile_image
bio
```

---

## 11-2. login_id 중복 검사 API

### 요청

```txt
GET /api/accounts/check-login-id/
```

### Query

```txt
?login_id=value
```

---

## 11-3. nickname 중복 검사 API

### 요청

```txt
GET /api/accounts/check-nickname/
```

### Query

```txt
?nickname=value
```

# 13. 이미지 처리 요구사항

## 프론트 처리

- 이미지 선택 즉시 미리보기 제공

- next/image 사용 가능

---

## 저장 방식

초기 구현:

```txt

```

향후 확장 가능:

```txt
S3 + CloudFront
Presigned URL 업로드
CDN URL 저장
```

# 19. 완료 기준

- 필수값 누락 시 회원가입 요청이 발생하지 않는다.
- login_id 자동 중복 검사가 동작한다.
- nickname 자동 중복 검사가 동작한다.
- 성공 시 초록 테두리 표시
- 실패 시 빨간 테두리 표시
- 실패 사유 텍스트 표시
- 회원가입 성공 시 로그인 페이지로 이동한다.
- 서버 에러가 사용자에게 표시된다.
- 프로필 이미지 미리보기가 정상 동작한다.
- 모바일 환경에서도 정상적으로 동작한다.
- 추후 PWA 확장이 가능한 반응형 구조를 유지한다.

```

```
